今回は、同じ長さを保って、隣同士で1文字ずつ異なる文字列が与えられる。

beginWord => ... => endWord

のように変換される最小の手順の数を求める。

endWordさえ含まれていれば、1文字ずつとなりが違うとわかっているのだから最大wordListの長さが手順の数ということ。

だけど実際は、1文字だけ違う文字列が隣同士以外にも存在する可能性があるのだから、wordList長さより短い道は見つかる。

問題を解剖すると、

- どの文字列からどの文字列にトランスメーション可能か
- どのトランスメーションの手順が最も早くendWordにたどり着くのか。

BFSで解いてみよう。

解いている中で、この前の隣接リストを作った問題のようにあらかじめ行き先の候補を計算しておく方法がある。

それがなんだかめんどくさいなと思ってしまって、その都度確認した方がよくない？と思ったので自分なりに違いを考える。

制約
- 文字列の長さ最大 `10`
- 文字列の個数最大: `5000`

- 先に隣接リストを用意しておく。
  - 文字列を1個ずつ比較していく必要があるから、
  - 10 * 5000^2 = 25 * 10^7 ≒ 3*10^8
  - これくらいのステップ数が必要。
- 行きついた文字ごとに、次の変換先候補を求める場合
  - これまで訪れた変換先は除外できる。
    一番最初はすべての変換先を調べなければいけないけれど、訪れるたびにその候補は減る。
  - endWordにたどり着いた場合、そこで計算が終わる。
  - ちょっとあまり定量的には書けなかったけどこれらの利点があると思う。

あと、BFSを最初whileでやろうと思っていたけど、探索の深さ、今回でいう変換手順数の記録がうまくできないと思ったので再帰関数を使った。

LeetCodeの解答だと、文字列の長さL、文字列の個数Nとして
- 候補となる文字列の作成 26char * L * N => O(N*L^2)
  - なんか解説自体にはO(N*L^2)が最悪計算量と書かれているが、たぶん違う。よくわからないことを言っている。
  引用
  > - Time complexity: O(m^2∗n)
  >   - M = length of each word, N = number of words in wordList
  >   - In the worst case, we might visit all words in the wordList
  >   - For each word, we try changing each of the M positions
  >   - For each position, we try all 26 letters
  >   - Creating each new word takes O(M) time due to string slicing: word[:i] + c + word[i+1:] ここが意味わからないけど、おそらく違う。
  >   - Set operations (lookup, add) are O(1) on average
  >   - O(N) words × O(M) positions × O(26) letters × O(M) string creation = O(M² × N)
- unordered_setでwordLisiに含まれるか 定数時間 O(1)
- よって、O(NL)


---

やっと解答自体の解き方について理解を始められる。

イメージとしては、
- beginSet、endSetがそれぞれの最前線。真ん中に近づいていく
- 二つの前線が重なれば、終了
- ここでいう前線は、次の変換先の候補の文字列の集合。
- 2つの前線をswapしているのは、候補が少ないほうが確認する文字列の数を少なくできるから。
  - 分かれ道の多いほうより少ないほうがいい。
  - だって、すべての道が正解でなくて行き止まりなこともある。

理解した後覚え書きできなくてんーってなってたけど、以下が理解できていなかった。
- 書き替えた1文字を戻す。
- endWordは入っている前提。
  そうじゃないと、endSetに最初にendWord入っていて、入ってなくてもそこからたどることが出来てしまう。
  双方向にするために最初から入れている。
