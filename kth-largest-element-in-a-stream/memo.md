# step1

単純に、降順のソートを考えてみる。

add自体の計算量は、標準ライブラリsortを使ったので、O(NlogN)になる。

最悪の計算ステップ数を考えると、多くて10^4回addが呼び出されるので、それを考えると
10^8になってくる。
やっぱりTLEした。

---

制約の一部

`0 <= nums.length <= 104`
`1 <= k <= nums.length + 1`


kの範囲の中だけでsortをすれば計算数も削減することが出来るかなと思ったが、うまくプログラムが書けなかった。 (step1-1)

具体的な考えとして、まずはk個の範囲で降順のソートをかけておいて、一番端(k個個目)と比べる。値が端より小さければ、ソートする必要もないので、今まで通りのk番目を返す。
そうでないなら、k番目の値を上書きしてソートかける。そしてk番目を返す。

というか制約を見ていて思ったのだけれど、addは必ず一回は呼ばれるのか。kの範囲がnums.length + 1も含んでいる。

この場合も結局、悪くて10^4のソートは書けないといけない。結局数倍程度の差か？

---

答えを見た。
Priority Queue、Heapとかいうやつを使うらしい。今回は特にカテゴリは気にせずに解いていた。
