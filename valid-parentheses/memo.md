# step 1

いきなり1つカッコ文字を与えられたときには、

1. かっこ閉じの場合 => 直前のかっこ開きの種類が何か知りたい。同じならOK
2. かっこ開きの場合 => 今このかっこ開きを扱っていることを(メモ 1.のために。)

stackを使ってかっこ開き文字を管理して、かっこ閉じのどれかが来たらそれで照合させていく手順でプログラムを書いてみる。

変な例えかもしれないけど、鍵付きドアが永遠に続いた迷路から脱出できるか？を考えてみる。
ルールとして、
鍵が見つかるまではとにかく次の部屋にすすまなければいけない。鍵が見つかれば、その鍵を使って前の部屋に戻れるかどうかを試す。

このとき、「鍵」と、前の部屋に戻るためのドアの「鍵穴」は、あることが条件である。
これはListでも再現可能？ただ、メモリの使用量的には、nextのポインタを保持してる分無駄？でもstackでもポインタは使われているように感じる。でもたしか、連続な領域で、topのポインタさえ保持していればよかった気がする。
実装が気になるけれど、難しそうだった。いつか読みたい。
処理の方向が決まってるからこそ、stackが活きてくる。

計算量は O(N)。
領域計算量は、最悪O(N) (すべてかっこ開きの文字が来た場合それだけstackに積み上がる);

最初、最後まで処理できていたらtrueだと思っていたが、open_brackets.empty()がtrueでないとそうはならなかった。
なぜなら、かっこ開きに対してかっこ閉じが足りない場合があるから。 "(("とか。
反対に、かっこ開きのほうが足りないこともある。単純にopen_brackets.empty()なのに、かっこ閉じがあればvalidでない。

あと、s[index] != '\n'の条件でwhileを回すように作っていたが、かならず改行文字があるわけでもないし、範囲for文のが簡単であることがわかったのでそれに書き換え。
