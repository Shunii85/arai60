# step1

アナグラムと呼ばれる、文字を入れ替えて作られた単語やフレーズがある。
与えられた複数の文字列を、アナグラムで分類する。

解法を2つ思いついた。

## 解法1

1. まずは、アルファベット順に並び替えさせる。
2. 同じ文字列を確かめていく。

文字列の個数をN、文字の長さをL(すべて同じではないが)とすると、
時間計算量が、ソートだけでO(NLlogL)になる。

ソートは最大おおよそ、`2*10^6`ステップほどかかる。
文字比較をする際に、O(N^2)くらい？ `10^8`かかる。ギリギリ行けるか？

100個の文字列だったら、シフトと組んで一人ひとり並べ替えるくらいは大差ない気がする。

ACはできた。

実装方法としては、
各文字列をソートして、それをキーとして、strsへのindexをvectorで複数格納できるようする。
そして、indexを使って、元の文字列を取り出して、順にグルーピング
文字列の比較は必要なくなった。

具体的に実行時間/メモリ使用量を計算してみたい。
- 計算量自体は、O(N * LlogL)
- ソート: `2*10^6`ステップ
- 解答のvectorを作るのに、`10^4`ステップ。文字列の個数分くらい。

合わせてだいたい、`2*10^6`ステップ。

C++で`1億ステップ(10^8)/秒 ~ 10億ステップ(10^9)/秒`実行できるとして、

`2ms ~ 20ms`

LeetCodeでもだいたいそんな感じになっていた。

メモリ使用量を考える。
- unordered_mapで、すべての文字列で別のアナグラムだった時、
  ( 最大100文字(100byte) + index1個 (int-4byte) ) * 10^4
  10^6byte程度
- 戻り値のためのvector
  100文字 * 10^4個 => 10^6byte程度
2 * 10^6程度
よって、4MB程度。

### ソースコードについて
vector::push_backにmoveで、右辺値参照を渡すようにした。これだと、ムーブ代入になるので、コピーが発生しない。

## 解法2

1. 文字列のなかにある文字が何個含まれているかメモする。
2. メモを見て、同じ個数文字が含まれていたら、グルーピングする。
